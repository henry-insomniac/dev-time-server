# 编码规范

## 适用范围

本文件定义 `dev-time-server` 后端代码规范和行数约束。技术栈、工具链和验证命令以 `tech-stack.md` 为准。

## 基础原则

- `dev-time-server` 管事实、权限、状态和确认后的 GitHub 写入。
- handler 保持薄层，业务规则进入 service 或领域包。
- 数据库访问通过 sqlc 生成代码或明确 repository 层，不在 handler 中拼 SQL。
- 风险评分规则必须可测试、可解释、可追溯到 RiskSignal。
- 所有外部写入必须幂等，并带权限校验和审计记录。

## Go 代码规范

- 所有 Go 代码必须通过 `gofmt`，导入排序使用 `goimports`。
- 包名短、小写、无下划线，表达领域而不是技术层泛称。
- 不创建泛用 `utils` 包；通用能力必须有明确调用边界和所有者。
- 函数返回 error 时必须携带上下文；对外响应映射为稳定错误码。
- context 从入口传入并贯穿调用链，不在业务函数中随意创建后台 context。
- 事务边界必须显式，函数名或调用点应能看出事务语义。
- 状态枚举使用稳定常量，禁止散落字符串。

## PostgreSQL 与数据访问

- GitHub raw payload 使用 `jsonb`，可查询字段提升为结构化列。
- migration 必须和代码变更一起提交，禁止只改运行库。
- 外键、唯一约束、状态约束由数据库兜底。
- 风险结果、AgentArtifact、ActionSuggestion 必须保留 evidence refs。
- 查询必须以访问路径为依据建立索引，避免先加索引再找场景。
- 后台 job 必须有状态、重试次数、错误摘要和幂等键。

## 行数规范

| 对象 | 目标上限 | 硬上限 | 处理方式 |
| --- | --- | --- | --- |
| Go 普通源文件 | 300 行 | 400 行 | 按领域、handler、service、repository 拆分 |
| 单个函数 | 60 行 | 100 行 | 提取校验、映射、事务步骤或规则函数 |
| HTTP handler | 80 行 | 120 行 | handler 只做解析、校验入口、调用 service、响应映射 |
| 风险规则文件 | 260 行 | 360 行 | 按风险类别拆分，例如 progress、blocked、quality |
| repository / query wrapper | 220 行 | 320 行 | 按聚合根或访问场景拆分 |
| 单元测试文件 | 350 行 | 500 行 | 按规则、状态机或 API endpoint 拆分 |
| 集成测试文件 | 400 行 | 600 行 | 按用户路径、GitHub 事件流或数据库场景拆分 |

超过硬上限时，PR 必须说明为什么暂不拆分，并补一个后续拆分任务。

## 例外

- `sqlc` 生成代码不受行数限制，但必须放在生成目录，不允许手改。
- migration 不受普通文件上限限制，但每个 migration 只表达一个清晰变更。
- 大型测试 fixture 不受普通行数限制，但必须放在 `testdata` 或明确 fixture 目录。

## 评审检查

代码评审时优先检查：

- handler 是否过厚。
- 事务、权限、幂等和错误映射是否明确。
- 风险评分是否仍由规则引擎负责，而不是 LLM 直接决定。
- GitHub 写入是否必须经过用户确认和服务端权限校验。
- 数据库约束是否覆盖关键一致性。
